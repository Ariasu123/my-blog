从 vLLM 和 SGLang 入门，到 AI Infra 求职落地

从 vLLM 和 SGLang 入门，到 AI Infra 求职落地

这两年，大模型工程岗位正在发生一个很明显的变化：

企业不再只需要“会调 API 的应用工程师”，而是越来越需要“会做推理系统的工程师”。

以前很多人做 AI 应用开发，只要能把模型接进产品、把接口跑通、把业务链路串起来，就已经算掌握了相当大的一块能力。但随着模型服务越来越深入生产环境，企业真正开始为下面这些问题付钱：

  • 吞吐能不能上去
  • 时延能不能压下来
  • 显存能不能扛住
  • 服务能不能稳定跑
  • 成本能不能控住
  • 高并发、长上下文、复杂生成场景下会不会出问题

这也是为什么 vLLM、SGLang、TGI、TensorRT-LLM 这类项目越来越受关注。

但如果你想从应用层转到 AI Infra，有一个判断一定要尽早建立起来：

vLLM 和 SGLang 不是学习终点，它们只是你进入 AI Infra 世界的入口。

真正决定你能不能走到求职落地那一步的，不是“你有没有把框架跑起来”，而是你有没有围绕它们建立起 推理系统理解、实验能力、排障能力和工程表达能力。

这篇文章想回答的，就是这个问题：

如果你想从应用层转到 AI Infra，应该怎么学 vLLM 和 SGLang，最后才能变成真正的求职竞争力？

先说结论：目标别定成“会部署框架”

如果你的目标是转向 AI Infra / LLM Serving / Inference Engineering，这条路线是成立的，而且在国内市场有真实岗位需求。

但很多人一开始就把目标定偏了。他们会觉得：

  • 把 vLLM 跑起来
  • 把 SGLang 启起来
  • 调通一个 OpenAI-compatible API

这样就算入门 infra 了。

这其实只摸到了门把手。

企业真正要的，不是“会起一个推理服务的人”，而是能把大模型推理系统 跑稳、跑快、跑大规模 的人。也就是说，框架只是切入口，真正有价值的是你围绕框架形成的系统能力。

企业真正看重的，是你能不能理解系统

如果把这个方向的能力压缩来看，其实就三层。

第一层：会跑服务

会启动 vLLM / SGLang、会配参数、会调接口，这当然是入门，但也只是入门。

它只能说明你接触过框架，不能说明你真正理解系统。

第二层：能理解系统、做实验、看指标

这才是求职的最低有效门槛，也是大多数转岗者最应该先达到的层级。

你至少要能讲清：

  • Prefill 和 Decode 的区别
  • KV Cache 为什么关键
  • 吞吐、时延、显存为什么互相拉扯
  • 长上下文为什么会改变系统行为
  • 连续批处理 / 动态批处理在解决什么问题

同时你还要能：

  • 设计一次压测
  • 采集关键指标
  • 对结果做解释
  • 判断问题大概出在哪一层

如果这一层做不到，面试时通常会非常被动。

第三层：能改框架、能排障、能做优化

这一层开始真正拉开差距。

你应该能看懂关键调用链，能在请求链路上增加日志或 metrics，能修改一个小的调度、缓存或观测逻辑，也能分析 OOM、P99 抖动、吞吐下降这类问题。

这还不一定意味着你已经是底层性能专家，但至少说明你已经不再停留在“会用框架”的阶段。

[!important]
对大多数从应用层转 infra 的人来说，最现实的目标不是一开始就做 CUDA / Kernel 优化，而是先稳定进入“第二层”，再逐步逼近“第三层”。

如何快速入门：先完成一个最小闭环

很多人一上来就想同时看 vLLM、SGLang、TGI、TensorRT-LLM，顺便再补 K8s、CUDA、Triton。这样看起来很努力，但结果通常是很快失去主线。

更好的入门方式，是先完成一个最小闭环：

跑起来一次 -> 看懂一点原理 -> 做一轮最小实验 -> 写下一页总结

这比你连续收藏十篇教程更有效。

第 1 步：先把最小服务跑起来

刚开始不要追求理解全部细节，先让自己获得一个基本手感：

  • 能启动一个 vLLM 服务
  • 能发起一次 API 请求
  • 能看到输入、输出、日志和资源占用
  • 能知道服务失败时大概去哪里看错误

这一步的目标不是“证明你会 infra”，而是先把抽象概念落到一个可运行的东西上。

第 2 步：马上回头补推理系统基础

跑通之后，不要停留在“能用了”。

你要立刻回头问几个问题：

  • 请求从哪里进来？
  • 为什么第一次生成前会有 Prefill？
  • 为什么后面是 token-by-token 的 Decode？
  • KV Cache 到底缓存了什么？
  • 为什么上下文变长之后，显存和时延都会变敏感？

这一步会让你从“调用者视角”切到“系统视角”。

第 3 步：做第一次小实验

不要等全部学完再做实验。最早的实验可以非常简单。

比如只改三个变量：

  • 输入长度
  • 输出长度
  • 并发数量

然后观察：

  • 请求变慢了吗？
  • 首 token 变慢了吗？
  • 输出阶段变慢了吗？
  • 显存变化明显吗？
  • 日志里有没有调度或 batch 相关信息？

哪怕实验很粗糙，也比单纯看文档更能建立直觉。

第 4 步：写一页入门总结

快速入门最后一定要写总结，否则很容易“跑过但没沉淀”。

这一页只需要回答 4 个问题：

  1. 我跑通了什么？
  2. 我理解了哪几个核心概念？
  3. 我做了哪几个变量实验？
  4. 我下一步最该看哪一段源码或文档？

做到这一步，你才算真正进入这个方向，而不是只是跑了一个 demo。

一个 7 天快速入门节奏

如果你想给自己一个明确起点，可以按下面这个节奏走：

时间 目标 产出
Day 1 跑通最小 vLLM 服务 一条成功请求 + 启动记录
Day 2 理解请求从 API 到生成的粗链路 一张请求流程草图
Day 3 补 Prefill / Decode / KV Cache 一页概念笔记
Day 4 改输入长度和输出长度做实验 一组简单对比数据
Day 5 改并发数观察吞吐和时延 一组并发实验记录
Day 6 看一段 scheduler 或 cache 相关代码 一段源码阅读笔记
Day 7 汇总“我跑通了什么、理解了什么、下一步看什么” 一页入门总结

这 7 天不追求完整，只追求建立第一轮闭环。完成之后，再进入系统学习会顺很多。

学习路线：不要铺太宽，要按 4 个阶段推进

如果你不是在做专项研究，而是想把这条路线真正走到求职落地，我更建议把它理解成 4 个阶段，而不是一堆并列知识点。

第 1 阶段：建立推理系统脑图

这一阶段的目标不是写很多代码，而是先把问题空间看清。

你需要回答的核心问题只有几个：

  • 一次请求是怎么进入系统并变成 token 流的？
  • 为什么 Prefill 和 Decode 的资源特征不一样？
  • 为什么 KV Cache 会直接影响吞吐、显存和时延？
  • 为什么长上下文、多轮对话、复杂生成会让系统行为变化很大？

这一阶段最好的产出不是“我看了多少页源码”，而是：

  • 一篇自己的推理系统基础笔记
  • 一张请求生命周期图
  • 一个最小 benchmark 脚手架

只要你能画出一条请求从进入服务到生成 token 的路径，你再看框架源码时就不会完全迷路。

第 2 阶段：围绕 vLLM 建立第一套系统理解

我建议先以 vLLM 为主框架。

不是因为它一定“最好”，而是因为它足够主流，也足够适合作为理解现代 LLM serving 的第一站。

这一阶段你应该围绕 vLLM 做四件事：

  • 跑通服务
  • 跟请求路径
  • 看 scheduler 和 cache 相关代码
  • 做小规模实验

注意，这里的重点不是把所有源码都啃完，而是建立“系统视角”：

  • 请求怎么被组织
  • batch 怎么被调度
  • cache 怎么被管理
  • 不同负载为什么会打出不同结果

只要你开始能用这些问题看 vLLM，你就已经不是在“学一个工具”，而是在学一套推理系统。

第 3 阶段：把 SGLang 学成“对照组”

很多人学第二个框架时，会重复第一遍的学习方式，结果是又学了一套命令，却没有形成判断力。

更好的方式是：把 SGLang 当成对照组。

你学习它，不是为了多背一套 API，而是为了比较：

  • 它和 vLLM 的设计思路哪里不一样
  • 它为什么在某些复杂生成场景更自然
  • 它适合什么负载，vLLM 又适合什么负载

一旦你带着“比较”和“选型”的视角去学，SGLang 才会真正变成你的面试资产。

最值得做的不是泛泛看文档，而是做一份小对比：

  • 同一个模型
  • 同一组请求
  • 同样记录吞吐、时延、显存和调试体验
  • 最后写出“什么时候选谁”的判断

第 4 阶段：把框架能力补成 Infra 能力

这一阶段是很多人最容易忽略，但企业最看重的一步。

因为到了求职现场，面试官要确认的不是“你会不会看框架”，而是：

  • 你能不能把服务真正跑起来
  • 你能不能容器化、观测、压测
  • 你能不能定位问题
  • 你能不能解释为什么这个问题会发生

所以你必须把这些能力补上：

  • Linux 基础命令和排障思路
  • Docker 镜像与容器运行
  • 基础 K8s 概念
  • 日志、指标、监控体系
  • GPU 监控
  • profiling 工具

这一阶段最好的产出，是一个带日志、指标和压测记录的最小 serving 系统。

只有走到这里，你的能力才会从“框架学习者”变成“infra 候选人”。

最后要沉淀成什么

如果你想让这条学习路线最终服务于求职，不要只留下阅读记录。

最少要沉淀两类东西。

第一类：实验报告

比如：

  • vLLM 在不同上下文长度下的表现
  • vLLM 和 SGLang 在相同请求下的差异
  • 并发变化对 TTFT / TPOT / 显存的影响

实验报告的重点不是数据多漂亮，而是你能解释：

  • 为什么这样设计实验
  • 看了哪些指标
  • 结果说明什么
  • 下一步应该怎么优化或验证

第二类：最小 serving 项目

这个项目不需要很大，但要完整。

至少包含：

  • 服务启动
  • API 调用
  • Docker 化
  • 基础日志
  • 基础指标
  • 一次压测记录
  • 一次问题分析

这比“我学过 vLLM / SGLang”有说服力得多。

最后的判断

如果你是为了找工作进入 infra，最稳的路线不是“赶紧把几个框架都跑起来”，而是：

先用 vLLM 跑通最小闭环，再补推理系统基础，然后围绕 vLLM 建立系统理解，用 SGLang 做对照，最后补齐 Linux / Docker / 观测 / profiling，并把过程沉淀成实验报告和 serving 项目。

从这个角度看，vLLM 和 SGLang 不是学习终点，而是你进入 AI Infra 世界最好的两扇门。
