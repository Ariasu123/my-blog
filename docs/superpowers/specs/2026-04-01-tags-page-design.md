# Tags Page Design

## Goal
在不影响首页、文章页、分类页与现有主题其余页面的前提下，重做标签页展示：

1. 用更现代的彩色块状标签替代默认文字云
2. 让每个标签拥有长期稳定、一致的专属颜色
3. 不显示文章数量
4. 将 front matter 中以 `/` 拼接的复合标签拆分后单独显示

## Scope
本次只处理标签页的展示生成逻辑与对应样式，不改动文章内容页、首页文章 meta、分类页、搜索结果展示或 Hexo 路由机制。

涉及范围：
- 项目侧标签页模板覆盖
- 标签页专用样式
- 如模板本身不足以保持逻辑清晰，则补充一个项目侧辅助脚本用于标签归一化与颜色生成

不在本次范围内：
- 批量回写或修改已有文章 front matter
- 重写 Hexo 标签归档系统
- 修改搜索索引逻辑，使其同步拆分复合标签
- 修改首页/文章详情中标签的展示规则
- 直接修改 `node_modules`

## Current Context
当前项目基于 Hexo + Fluid。
- 标签页入口为 `source/tags/index.md`
- Fluid 默认标签页模板使用 `tagcloud()` 生成文字云
- 默认实现会按主题配置的字号与渐变色渲染标签，不适合当前“固定色块、独立分块”的目标
- 用户新增约束：部分 front matter 中的标签以单个字符串形式存在，但内部用 `/` 连接多个概念；标签页上必须把这类值拆成多个单独标签显示

## Approved Constraints
- 标签页必须从默认文字云改为独立色块/卡片式展示
- 每个标签颜色必须固定且长期一致，不能每次构建随机变化
- 标签块上不显示文章数量
- 像 `AI/CUDA/推理优化` 这样的复合标签，在标签页中必须拆成 `AI`、`CUDA`、`推理优化` 三个独立标签显示
- 本轮只保证标签页展示正确，不顺带扩散到首页、文章页、搜索索引或底层标签归档数据模型
- 不修改 `node_modules`

## Approach Options

### Option A - Recommended: Project-side tags template override with display-layer normalization
在项目侧覆盖 `tags.ejs`，自行构建标签页数据：
- 先读取原始标签集合
- 在标签页展示层对带 `/` 的标签名做拆分归一化
- 用稳定算法为每个拆分后的标签生成固定颜色
- 渲染为响应式色块列表

Pros:
- 改动集中在标签页，侵入最小
- 最符合当前需求边界
- 不依赖修改第三方主题源码
- 可明确控制“拆分只发生在标签页展示层”

Cons:
- 需要在模板内或辅助脚本中补一小段标签归一化逻辑
- 标签页展示结果会与底层原始 tag 对象存在一定抽象差异

### Option B - Normalize all tags at content or data layer
在文章 front matter 或 Hexo 更底层的数据层统一拆分复合标签，再让所有页面共享这套结果。

Pros:
- 数据语义最统一
- 未来其他页面也能自然受益

Cons:
- 改动面过大
- 会影响文章页、搜索、标签归档等现有行为
- 不符合本轮“小步迭代、最小侵入”的边界

### Option C - Style the existing tagcloud only
保留默认 `tagcloud()` 输出，只通过 CSS 把文字云勉强改成块状外观。

Pros:
- 改动最少

Cons:
- 无法可靠满足“固定颜色”“不显示数量”“拆分复合标签”这三个核心要求
- 仍受默认标签云数据结构限制

结论：采用 **Option A**。

## Design

### 1. Rendering Strategy
标签页不再调用默认 `tagcloud()`，改为项目侧自定义模板渲染。页面主体将输出一个响应式标签网格/流式块列表，每个标签都是独立可点击元素。

每个元素包含：
- 拆分后的单个标签名
- 指向对应标签页的链接（若存在标准标签归档地址则复用）
- 稳定生成的背景色与前景色

页面不展示文章数，也不再通过字号大小表达权重；视觉重点从“频次”改为“标签本身的可读性与浏览体验”。

### 2. Tag Normalization
标签页展示阶段会对原始标签名做归一化处理：
- 若标签名不包含 `/`，则直接作为一个独立标签展示
- 若标签名包含 `/`，则按 `/` 拆分为多个子标签
- 拆分后会去除两端空白，并忽略空字符串
- 相同结果标签会合并，避免页面出现重复块

例如：
- `CUDA` → `CUDA`
- `AI/CUDA/推理优化` → `AI`、`CUDA`、`推理优化`
- `AI / CUDA / 推理优化` → `AI`、`CUDA`、`推理优化`

本次归一化只服务于标签页展示，不回写文章 front matter，也不强行改变 Hexo 其他页面的数据来源。

### 3. Color Strategy
颜色必须“同名稳定、长期一致”。实现上使用一个确定性映射方案：基于标签文本生成稳定哈希，再映射到一组受控的色相/明度/饱和度区间，保证：
- 同一个标签在不同构建中颜色一致
- 不同标签通常具有可区分的颜色
- 整体视觉仍落在当前博客风格允许的柔和彩色范围内

为了保证可读性，文字颜色不跟随随机变化，而是使用与背景对比度足够的固定深色或按亮度规则选择深浅前景色。

### 4. Layout and Styling
样式只作用于标签页。

视觉目标：
- 标签块具有统一圆角、留白和悬停反馈
- 页面在桌面端呈多列/流式排布，在移动端自然换行
- 颜色丰富但不过度刺眼，与当前站点蓝白调性兼容
- 不使用文章数量徽标，不引入额外说明文字负担

样式预计放在 `source/assets/css/custom.css`，通过标签页专用选择器约束，避免影响其他页面。

### 5. File-Level Impact
预计涉及以下文件：
- `source/tags/index.md` - 保持标签页入口，不需要结构性修改
- 项目侧 `layout/tags.ejs` 或等价覆盖文件 - 替代默认文字云模板
- `source/assets/css/custom.css` - 新增标签页专用样式
- 如有必要，新增一个项目侧脚本文件，为模板提供标签拆分与颜色计算辅助逻辑

## Data Flow
本次标签页数据流预计为：
1. Hexo 提供原始标签集合
2. 项目侧标签页模板读取标签集合
3. 模板或辅助脚本对标签名做拆分归一化
4. 归一化后的标签生成稳定颜色
5. 模板输出可点击色块列表

改动只介入标签页渲染阶段，不改 Hexo 站点的全局标签数据结构。

## Error Handling
- 若某个标签名拆分后出现空段（如连续 `/`），应直接忽略空结果，避免渲染空白标签块
- 若多个原始标签拆分后得到相同标签名，应合并显示，避免重复
- 若颜色计算逻辑失效，应回退到一组安全默认色，而不是让页面失去可读性
- 若项目侧模板覆盖未生效，页面最多退回默认文字云，不影响站点其他页面

## Validation Plan
完成实现后按以下方式验证：

1. 构建站点并打开标签页
   - 页面不再是默认文字云
   - 标签以独立彩色块形式展示
2. 检查复合标签样例
   - `AI/CUDA/推理优化` 不再以整串形式出现
   - 页面中分别出现 `AI`、`CUDA`、`推理优化`
3. 检查颜色稳定性
   - 连续两次构建后，同一标签颜色保持一致
4. 检查展示约束
   - 标签块不显示文章数量
   - 点击标签仍可进入对应标签页或既有归档链接
5. 回归其他页面
   - 首页、关于页、文章页不因本轮修改发生布局回归

## Out of Scope Notes
如果后续希望把“复合标签拆分”升级为全站统一的数据语义（包括文章页标签、搜索索引、标签归档页统计等），应单独开新一轮 spec，处理更底层的数据标准化，而不是在本轮标签页美化中顺手扩展。
