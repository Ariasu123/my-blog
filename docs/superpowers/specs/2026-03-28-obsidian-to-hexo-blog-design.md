# Obsidian-to-Hexo Automated Blog 设计稿

日期：2026-03-28
范围：本轮仅细化 Repo B（公开博客构建仓库），Repo A 先保留输入契约与后续扩展边界。

## 1. 背景与目标

本项目要在当前仓库中构建一个基于 Hexo 的个人博客，并通过 GitHub Pages 的 `gh-pages` 分支对外发布。整体系统采用双仓架构：

- **Repo A**：私有内容源仓库，承载 Obsidian Markdown，决定哪些文章可发布。
- **Repo B**：公开博客构建仓库，接收可发布文章、构建静态站点、部署到 GitHub Pages。

第一版博客的设计目标：

1. 站点定位为**内容优先的个人博客**，首页以文章列表为核心。
2. 功能结构参考 `https://www.aivi.fyi/`，但不复制其视觉样式。
3. 第一版必须包含：**分类/标签、站内搜索、关于页/社交入口、独立项目页**。
4. 首页文章列表默认展示：**封面图、标题、摘要、日期、分类/标签、阅读时长**。
5. 使用 `MapleMono-NF-CN-Regular.ttf` 增强技术博客气质。
6. 第一版通过 **默认 github.io 地址** 上线，不接自定义域名。
7. 实现策略必须支持**后期更换主题**，因此坚持低侵入定制。
8. 最终产物需整理为一个可直接上传到 GitHub 的标准仓库版本，并由 Agent 在实现完成后协助创建并推送到用户的 GitHub 仓库，便于后续代码管理。
9. GitHub Pages 采用**项目站点仓库**形态，目标仓库名暂定为 `my-blog`，部署路径按 `https://<用户名>.github.io/my-blog/` 预留配置。

## 2. 非目标

第一版明确不做以下内容：

- 自定义域名接入与 DNS 配置
- GitHub 组织级权限、团队协作权限或复杂仓库治理流程
- 服务端搜索或任何后端服务
- 自动生成复杂项目卡片数据源
- 复杂图片托管编排或内容清洗流水线
- 深度魔改主题导致的强耦合实现
- Repo A 的完整实现细节（本轮只定义接口）

## 3. 站点信息架构

### 3.1 页面结构

第一版站点固定包含以下页面：

1. **首页**
   - 主体为最新文章列表
   - 每篇文章展示封面图、标题、摘要、日期、分类/标签、阅读时长
   - 作为站点主入口与持续更新入口

2. **文章详情页**
   - 展示正文、封面、元信息
   - 支持分类/标签跳转
   - 代码块使用 MapleMono 字体

3. **分类页**
   - 按分类聚合文章

4. **标签页**
   - 按标签聚合文章

5. **搜索页 / 搜索入口**
   - 使用前端本地搜索，适配纯静态站点

6. **关于页**
   - 展示个人简介、技术方向、联系方式、社交链接

7. **项目页**
   - 独立页面
   - 采用**项目卡片列表**组织内容
   - 每个项目卡片至少包含：项目名称、简介、技术栈、链接入口
   - 内容以手工维护为主，不做自动聚合

### 3.2 导航结构

顶部导航保持轻量，第一版建议固定为：

- 首页
- 分类
- 标签
- 项目
- 关于
- 搜索

若所选主题支持侧边栏，则侧边栏承载轻量个人信息：

- 头像
- 一句简介
- 社交链接
- 少量站点说明

## 4. 核心技术方案

### 4.1 技术栈

- **静态站框架**：Hexo（最新版）
- **包管理器**：npm
- **部署方式**：GitHub Pages，发布目标为 `gh-pages` 分支
- **实现策略**：成熟主题二次定制 + 低侵入扩展

### 4.2 低侵入定制原则

为保留后期换主题能力，所有定制遵循以下边界：

1. 优先使用主题提供的配置能力。
2. 自定义页面尽量作为独立 page 实现，而不是塞进主题内部模板。
3. 样式优先通过覆盖样式层实现。
4. 仅在必要时改少量主题模板。
5. 不将封面同步、项目页内容、搜索入口等核心逻辑写死在单一主题内部结构里。

这意味着未来切换主题时，主要迁移对象应为：

- 根目录 `_config.yml`
- 主题专属覆盖配置
- 独立页面内容（如关于页、项目页）
- 自定义脚本（如 `scripts/sync-covers.js`）
- 少量覆盖样式

而不是重做整站内容系统。

### 4.3 主题选择约束

实现阶段应选择一个成熟 Hexo 主题作为底座，至少满足：

- 支持博客首页文章流
- 支持分类与标签页面
- 支持独立页面
- 允许加入本地搜索能力
- 允许添加侧边栏或顶部个人信息区
- 易于追加覆盖样式与少量模板调整
- **必须原生支持文章封面（Cover/Thumbnail）和阅读时长（WordCount/Reading Time）的配置开启，严禁通过深度覆写核心模板来强行实现**

推荐备选主题包括：**Fluid、Butterfly、NexT 的特定 scheme**。

本设计不预先锁定具体主题名称，但后续实现计划必须先完成主题筛选与验证，再开始具体接入。

## 5. 内容输入与双仓边界

### 5.1 Repo A → Repo B 输入契约

虽然本轮不实现 Repo A，但需要先定义接口：

- Repo A 遍历 Markdown 文件
- 解析 YAML Front-matter
- 仅筛选 `published: true` 或 `published = true` 的内容
- 将合格 Markdown 推送到 Repo B 的 `source/_posts/`

### 5.2 职责分离

- **Repo A**：内容源与发布状态控制
- **Repo B**：展示层、构建层、部署层

Repo B 不负责决定某篇文章是否发布，只负责接收、处理、构建与发布。

## 6. Repo B 目录结构设计

第一版建议采用如下结构：

- `source/_posts/`：同步进来的文章 Markdown
- `source/projects/index.md`：独立项目页入口
- `source/about/index.md`：关于页入口
- `source/assets/fonts/MapleMono-NF-CN-Regular.ttf`：字体文件
- `scripts/sync-covers.js`：封面同步脚本骨架
- `.github/workflows/deploy.yml`：部署工作流
- `_config.yml`：站点主配置
- `_config.<theme>.yml`（如主题支持）：主题覆盖配置

Agent 需在 `_config.yml` 中根据部署形态正确配置 `url` 和 `root`，确保相对路径资源加载正常。当前按项目站点仓库形态预留：仓库名 `my-blog`，访问路径目标为 `https://<用户名>.github.io/my-blog/`。

如所选主题对独立 page 路径有特定要求，可在不改变语义的前提下做小幅调整，但必须保留“项目页/关于页为独立页面”的设计结论。

## 7. 文章与项目内容展示设计

### 7.1 首页文章流

首页文章列表第一版默认展示：

- 封面图
- 标题
- 摘要
- 日期
- 分类/标签
- 阅读时长

实现上优先使用主题现成能力；若主题默认不展示某项元信息，可通过轻量插件、配置或少量模板扩展补齐，但不得演变为整站深改。

### 7.2 项目页

项目页是独立页面，不与首页文章流混排。

第一版采用**项目卡片列表**：

- 每个卡片包含项目名称、简介、技术栈、链接
- 内容由用户手工维护
- 页面排序由用户控制
- 第一版不引入单独项目内容模型、不自动从文章或数据文件生成卡片

这样可以保证：

- 页面结构稳定
- 后期换主题时迁移成本低
- 项目展示不干扰博客首页的“文章优先”定位

## 8. 搜索方案

第一版使用**前端本地搜索**：

- 构建时生成搜索索引
- 前端页面直接读取索引完成搜索
- 不依赖数据库或服务端
- 适配 GitHub Pages 纯静态托管
- 需在 `package.json` 中引入类似 `hexo-generator-searchdb` 的官方或主流插件，以确保 `npm run build` 时能自动在 `public/` 目录下生成搜索索引文件

搜索范围至少覆盖：

- 标题
- 摘要 / excerpt
- 正文基础内容
- 标签或分类（若插件能力支持）

若主题自带兼容的本地搜索方案，优先复用；若无，则引入轻量搜索插件并保持与主题低耦合。

## 9. 字体接入方案

项目必须使用以下字体资源：

- `source/assets/fonts/MapleMono-NF-CN-Regular.ttf`

接入要求：

1. 通过主题可覆盖的 CSS / Stylus / 自定义样式入口声明 `@font-face`。
2. 第一版优先将字体用于：
   - 代码块
   - 行内代码
   - 可选标题强调区域
3. 第一版**不强制整站正文全局使用 MapleMono**，避免长文阅读体验变硬。
4. 生成产物中的字体路径必须正确指向 `assets/fonts/...`，并兼容 GitHub Pages 路径环境，避免硬编码为仅适用于单一路径结构的绝对地址。

## 10. 脚本工程规范

`package.json` 必须包含以下脚本：

```json
{
  "scripts": {
    "cover:sync": "node scripts/sync-covers.js",
    "server": "npm run cover:sync && hexo server",
    "build": "npm run cover:sync && hexo clean && hexo generate"
  }
}
```

### 10.1 `cover:sync` 职责边界

`scripts/sync-covers.js` 第一版只承担基础职责：

- 扫描 `source/_posts/` 下的文章
- 读取 front-matter
- 对封面字段做统一化或基础同步预处理
- 输出明确日志，便于排查问题

第一版不将其扩展为复杂内容加工流水线。

### 10.2 本地与 CI 统一入口

- `npm run server`：本地预览入口，内置封面同步
- `npm run build`：CI 构建入口，内置封面同步 + Hexo 完整构建流程

这样可以保证本地预览与 GitHub Actions 构建尽量使用同一执行链路，降低环境偏差。

## 11. CI/CD 设计

### 11.1 Repo B 部署流程

Repo B 的 `.github/workflows/deploy.yml` 必须满足：

1. 监听 `main` 分支 `push`
2. 执行 `actions/checkout`
3. 安装指定 Node.js 环境
4. 执行 `npm install`
5. 严格执行 `npm run build`
6. 使用 `peaceiris/actions-gh-pages` 或同类标准方案，将 `public/` 发布到 `gh-pages`
7. 构建配置需兼容 GitHub Pages 项目站点仓库形态，确保发布后资源路径与页面路由可在 `my-blog` 子路径下正常工作

### 11.2 构建链路

主链路固定为：

`main 分支内容更新`
→ `GitHub Actions 拉取代码`
→ `npm install`
→ `npm run build`
→ 生成 `public/`
→ 发布到 `gh-pages`

## 12. 错误处理原则

第一版遵循“尽早暴露错误”的工程原则：

- `npm run build` 失败时，CI 必须直接失败
- `sync-covers.js` 发现关键字段异常时，应输出清晰日志，不静默吞掉错误
- 字体路径错误应能在构建产物检查中暴露
- 搜索索引生成失败时，应给出明确报错或可识别的降级行为

## 13. 验证与验收

### 13.1 本地验证

至少验证以下命令：

- `npm install`
- `npm run server`
- `npm run build`

验证点：

- 本地服务可启动
- `public/` 成功生成
- 首页、文章页、分类页、标签页、项目页、关于页可访问
- 搜索可使用
- 字体资源路径正确

### 13.2 构建产物验证

重点检查：

- `public/` 目录生成完整
- CSS 中字体路径正确指向 `assets/fonts/...`
- 首页文章列表完整展示封面、标题、摘要、日期、分类/标签、阅读时长

### 13.3 CI 验证

需满足：

- `main` 分支新增或更新 Markdown 后可触发部署工作流
- 工作流成功执行 `npm run build`
- 构建产物成功推送到 `gh-pages`

### 13.4 第一版验收标准

第一版完成时，必须同时满足：

1. 站点可通过默认 `github.io` 地址访问
2. 首页为文章优先布局
3. 支持分类、标签、站内搜索、关于页、独立项目页
4. 首页文章列表展示封面、标题、摘要、日期、分类/标签、阅读时长
5. MapleMono 字体已正确接入
6. `cover:sync`、`server`、`build` 三个脚本可用
7. `deploy.yml` 可自动完成 GitHub Pages 发布
8. 代码仓库结构适合直接推送到 GitHub，便于持续维护
9. 实现完成后可协助创建名为 `my-blog` 的 GitHub 仓库并推送代码
10. 整体实现保持低侵入，后期可替换主题

4. 本地搜索方案选型

6. `sync-covers.js` 骨架实现

10. 本地与 CI 验证路径
